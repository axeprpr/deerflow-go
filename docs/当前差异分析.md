# deerflow / deerflow-go 当前差异分析

本文基于当前 `deerflow-go` 代码状态，重新判断三类差异：

1. 接口层
2. 结构层
3. 行为层

结论不再沿用旧阶段的判断，而是按当前代码和当前回归结果重算。

## 一、总体结论

当前状态可以概括成三句话：

- 接口层：已经接近“主前端兼容完成”，剩余主要是边角和额外兼容面
- 结构层：已经比早期明显收敛，但仍然不是原版那种 `Gateway + Harness` 原生分层
- 行为层：核心聊天/工具链路已经明显向原版收敛，但仍受模型、provider、middleware 完整度影响

所以当前最准确的定位不是：

- “已经完全等同原版”

而是：

- “外部行为和主接口面已经很接近，内部结构和运行时实现仍然是 Go 版自己的路线”

## 二、接口层基线

详细列表见：

- [接口误差](./接口误差.md)

这里给判断结论。

### 1. 主前端路径已基本闭合

目前原版前端依赖的主路径：

- models
- skills
- agents
- memory
- mcp
- user-profile
- uploads / artifacts
- channels
- suggestions
- `/api/langgraph/*`

在当前 `deerflow-go` 上都已经有可用实现。

并且，这几轮已经对齐了：

- strict LangGraph run 校验
- skills install/update 返回体
- agent name / mutation error detail
- user-profile / mcp / memory / uploads / channels 的错误契约
- plain `/threads/*` 和 `/runs/*` 兼容面的 JSON `detail`
- stream/join/replay 在进入 SSE 之前的错误契约

### 2. 当前接口层的真实剩余差异

现在接口层剩下的，不再是“主前端会坏”的差异，而是：

- `deerflow-go` 暴露了原版没有的额外兼容接口
- 一些自扩展接口没有上游对应物，只能做风格统一，不能做完全同构
- 少量低频边角返回体/文案仍可能继续发现

因此现在更准确的判断是：

- 接口层已经从“功能补齐阶段”进入“边角维护阶段”

## 三、结构层基线

这部分仍然是两边最大的实质差异。

### 1. 原版结构

原版 DeerFlow 的核心边界是：

- `frontend/`
  - Next.js UI
- `backend/app/gateway`
  - FastAPI Gateway
- `backend/packages/harness/deerflow`
  - 真正的 runtime / agent / middleware / memory / sandbox

关键点不在文件多少，而在边界：

- Gateway 是应用层
- Harness 是运行时层
- App 可以依赖 Harness
- Harness 不反向依赖 App

这也是上游测试里明确保护的边界。

### 2. 当前 Go 版结构

当前 Go 版已经不再是早期那种“所有事情都塞在 `pkg/langgraphcompat`”的状态。

现在的主结构是：

- `pkg/langgraphcompat`
  - HTTP 路由
  - 协议兼容
  - Gateway 兼容视图
  - SSE 适配
- `pkg/harness`
  - factory
  - runner
  - lifecycle
  - feature assembly
  - sandbox provider 抽象
- `pkg/harnessruntime`
  - preflight
  - orchestrator
  - context
  - coordination
  - completion
  - outcome
  - query
  - memory runtime
  - lifecycle/runtime config

也就是说，这一轮重构后已经建立了更清楚的边界：

- compat 层主要处理协议和 HTTP
- harness/harnessruntime 承担运行时装配和运行时服务

### 3. 当前结构层相比旧阶段已经明显收敛

这几轮结构改动收掉了很多旧问题：

- `threads.go` 退出主结构
- run/thread/memory/gateway admin/runtime 已拆成独立 handlers/service
- runtime assembly 不再直接堆在 compat 里
- memory/summarization/title/clarification 已进入 lifecycle feature 路线
- runtime adapter 已按 concern 拆分
- run preflight / state / coordination / completion / outcome 已下沉到 `pkg/harnessruntime`

这说明：

- `deerflow-go` 现在已经不是单纯的“大 compat 包”
- 已经出现了接近上游 `Gateway -> Runtime` 的单向边界

### 4. 但结构层仍未完全等同原版

即便如此，当前结构仍和原版有本质差异：

#### 原版

- Gateway 和 Harness 是天然分开的两层
- 真实 agent runtime 是 LangChain / LangGraph
- memory/sandbox/middleware 都是原生 runtime 子系统

#### Go 版

- 仍然是单服务
- compat 和 runtime 虽已分层，但仍共处同一进程、同一仓库实现线
- `langgraphcompat` 仍然持有一部分上游没有的兼容状态和适配职责

所以当前结构层更准确的判断是：

- **已经从“明显混杂”进入“分层收敛中后期”**
- **但仍不是原版那种天然多层架构**

## 四、行为层基线

这里要分开看。

### 1. 核心执行路径已经明显收敛

这几轮已经收掉的关键行为差异包括：

- tool 顺序与工具面
- `present_files` 契约和时序
- `read_file` 默认截断
- clarification prompt 与技能读取路径
- `recursion_limit` / turn limit
- run disconnect / continue / join 语义
- stream replay / join / completion 收尾
- artifact / outputs 路径
- 具体请求直接执行、模糊请求追问 这条核心边界的 live 回归

目前这两条 live 行为回归已经稳定存在：

- 具体请求：
  - `帮我生成一个小鱼游泳的页面`
  - 直接执行，不走 `ask_clarification`
- 模糊请求：
  - `帮我做一个页面`
  - 追问更多细节，不直接产出

这意味着行为层的主问题，已经不再是“方向完全不对”。

### 2. 行为层仍不是完全等价

当前行为层剩余差异，主要在这几类：

#### 模型 / provider 敏感性

之前已经验证过：

- 同一条提示词，换 provider 或换模型，原版自己都会发生路径变化
- 例如 MiniMax 会让原版本身也更倾向先 clarification

这意味着：

- 行为对齐不能脱离“使用哪一个模型/provider”的前提

#### middleware 丰富度差异

原版是真正的：

- `create_agent(...)`
- middleware pipeline

并且有完整的：

- clarification
- loop detection
- memory
- title
- todo
- summarization
- token usage
- tool error handling

Go 版虽然已经把不少能力收进 lifecycle / features，但运行时内核仍然是：

- 自定义 ReAct loop
- Eino 负责模型/协议层

所以它在行为层上会越来越像，但内在触发点和时序仍不是原版完全同一套。

#### memory / sandbox 对行为的间接影响

用户这次要求“抛开 memory、sandbox 等执行层面”，这是合理的。

但要注意：

- 原版 memory middleware
- 原版 sandbox provider
- 原版 uploads / view-image / tool middleware

都会间接影响最终模型上下文和工具决策。

所以严格说：

- 即便主提示词和工具顺序已经对齐
- memory/sandbox/middleware 的差异仍可能继续影响长链路行为

## 五、按三层重新给结论

### 接口层

- 已经接近完成
- 剩余主要是边角和额外兼容面
- 不再是当前最大的风险源

### 当前 issue 回归基线

按当前 `docs/upstream-issue-regression-cases.md` 的状态看：

- 已覆盖：
  - `P0-1` Skill 可发现且可触发
  - `P0-2` Tool-call 连续性不丢失
  - `P0-3` 长 CSV 分析不重复执行已完成步骤
  - `P1-1` 联网搜索工具可用（能力面）
  - `P1-2` 模型映射和 provider 错误可诊断
- 未覆盖：
  - `P2-1 invalid_signature`
  - `P2-2 切换聊天后内容为空`

最近一轮正式对比（`qwen3.5-27b`）结果：

- `core`：go `12/12`，upstream `11/12`
- `extended`：go `10/10`，upstream `10/10`

本轮新增收口点：

- 上传文件自动生成的 markdown 伴生文件默认不进入线程 `artifacts`（可用 `DEERFLOW_ARTIFACT_INCLUDE_UPLOAD_MARKDOWN=true` 打开旧行为）
- `C14`（上传后同线程总结+动作）从 `FAIL` 收口到 `PASS`
- `C07/C08/C09` 的 artifacts 行为与上游更一致（不再被上传伴生 markdown 污染）

这里最重要的判断是：

- 当前剩余可见差异已主要集中在 **shape mismatch**（是否先追问、是否额外调用某个工具），而不是功能不可用
- 功能正确性（pass/fail）层面，go 端在现有回归集上已稳定
- 行为完全同构仍受模型/provider 选择影响（尤其是 clarification 触发和工具路径偏好）

这说明后续优化重点应该从“修功能失败”切换到“收敛行为形态”：

1. clarification 触发阈值收敛（减少与上游的形态偏差）
2. URL 明确场景下 `web_fetch/web_search` 的路径收敛
3. 模型分层策略继续沉淀（structured/stream tool-call 策略）

而不是继续扫接口小边角。

### 结构层

- 已明显收敛
- 但仍然是当前和原版最实质的架构差异来源
- `harness/harnessruntime` 已经把方向立起来了，但还不是原版天然分层

### 行为层

- 主路径已经明显收敛
- 但还没有达到“可忽略模型/provider/middleware差异”的程度
- 后续复盘应该重点看：
  - 哪些行为差异真的是代码问题
  - 哪些只是模型/provider 差异
  - 哪些是 memory/sandbox/middleware 侧效应

## 六、下一步建议

当前更合理的顺序不是继续机械扫小接口，而是：

1. 以当前代码为基线，重新列一份“剩余真实差异”清单
   - 接口层
   - 结构层
   - 行为层
2. 行为层只盯可复现、用户可感知、可与原版稳定对比的用例
3. 结构层继续围绕 `compat -> harness -> runtime` 单向边界收尾，而不是再回到大文件堆逻辑

如果只用一句话概括当前状态：

- **接口层已接近收口，结构层进入中后期收敛，行为层已明显接近原版但仍未完全等价。**
