# 接口误差

本文只讨论当前 `deerflow-go` 与原版 DeerFlow 的接口层差异，不讨论内部结构。

结论先说：

- 原版前端当前依赖的主接口面，`deerflow-go` 已经基本覆盖。
- 过去那种“缺路由、直接 404/405”的主问题已经基本收掉。
- 当前剩余差异，主要是少量非主路径边角、部分自扩展接口、以及 `deerflow-go` 额外暴露的兼容接口。

## 对比基线

本次判断基于：

- 原版前端调用面
  - `frontend/src/core/models/api.ts`
  - `frontend/src/core/skills/api.ts`
  - `frontend/src/core/agents/api.ts`
  - `frontend/src/core/mcp/api.ts`
  - `frontend/src/core/memory/api.ts`
  - `frontend/src/core/uploads/api.ts`
  - `frontend/src/core/threads/hooks.ts`
  - `frontend/src/core/api/api-client.ts`
- 原版 Gateway 路由
  - `backend/app/gateway/routers/*.py`
- 当前 Go 版 compat 实现
  - `pkg/langgraphcompat/*.go`

## 当前状态

### 1. 主前端接口面已基本对齐

当前已经确认可覆盖的主路径包括：

- `/api/models`
- `/api/skills`
- `/api/agents`
- `/api/user-profile`
- `/api/mcp/config`
- `/api/memory`
- `/api/channels`
- `/api/threads/{thread_id}/uploads`
- `/api/threads/{thread_id}/artifacts/{artifact_path...}`
- `/api/threads/{thread_id}/suggestions`
- `/api/langgraph/*`

这意味着：

- “直接接原版前端能否跑起来”这一层，当前答案是可以。
- 接口问题已经从“缺实现”进入“收边角”和“逐项契约收口”阶段。

### 2. 这几轮已经收掉的关键误差

当前代码已经对齐了以下高频差异：

- `/api/langgraph/*` run 请求严格校验
  - `assistant_id` 必填
  - `thread_id` 在严格 LangGraph run 路径上要求合法 UUID
- `PUT /api/skills/{skill_name}` 返回体
  - 现在直接返回 skill 对象
- `POST /api/skills/install` 返回体
  - 现在只保留前端依赖的字段
- `agents` 非法 name 与 mutation 失败 detail
- `user-profile` 读写失败 detail
- `mcp/config` 更新失败 detail
- `memory` 的 clear / reload / delete fact / import 失败 detail
- `thread delete / uploads / channel restart` 的 HTTP 契约
- plain `/threads/*` 与 `/runs/*` 兼容面的 JSON `detail`
- stream/join/replay 在进入 SSE 之前的错误契约
- 线程 `artifacts` 默认不再混入上传文件的 markdown 伴生文件（更贴近原版 UI 契约）

## 当前仍然存在的真实误差

### 1. `deerflow-go` 仍然暴露了上游没有的额外兼容接口

这不是 bug，但它说明两者不是“完全同构”：

- plain `/threads/*`
- plain `/runs/*`
- `/api/memory/export`
- `/api/memory/import`
- `/api/memory/facts`
- `/api/memory/facts/{fact_id}` `PATCH`
- `/api/tts`
- token auth 兼容层

这些接口对兼容和自托管有价值，但它们不属于“原版前端主接口面”。

### 2. 一些自扩展接口没有原版一一对应物

例如：

- `memory import/export/facts patch`
- `tts`
- auth token 保护

这类接口无法做“逐条和原版完全一致”的机械对齐，因为原版本身没有同等主路径。

更合理的标准是：

- 风格统一
- 错误结构统一
- 不影响原版前端主路径

### 3. 仍可能存在少量低频返回体或错误文案边角

现在的差异已经不再集中在主 UI 路径，而更像：

- 很低频的错误 detail 用词
- 自扩展接口的非阻塞差异
- 成功返回里的附加字段

这些问题仍值得继续收，但优先级已经明显低于前几轮。

## 当前判断

更准确的结论应该是：

- 当前接口层已经达到“主前端可用且主要契约已收口”的状态。
- 现在不能再说“接口层完全一样”。
- 但也不能再说“接口层差异很大”。
- 剩余误差主要是：
  - 额外兼容接口
  - 自扩展接口
  - 少量低频边角契约

## 建议

如果目标是“继续提升对上游的一致性”，下一步顺序建议是：

1. 先做结构层与行为层复盘，不再继续机械扫小接口
2. 接口层只在复盘中发现“前端真实可感知”的残余误差时再补
3. 对无原版对应物的自扩展接口，改用“统一风格”而不是“伪装成完全等同原版”的标准
